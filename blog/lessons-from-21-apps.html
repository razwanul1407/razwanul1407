5 Lessons From Building 21+ Mobile Apps

Md Razwanul Hasan

February 28, 2025  ·  7 min read

I started coding mobile apps in 2019 with no idea what I was doing. Six years, five companies, and 21+ shipped apps later, I've learned things the hard way that I wish someone had told me on day one.

Lesson 1: Scope Creep Kills Projects — Protect It Aggressively

The most dangerous moment in any project is when a client says "while you're at it, can we also add...". I've seen projects double in scope mid-development, blow past deadlines, and end with frustrated clients and burned-out developers.

What I do now: Write a Feature Specification Document at the start of every project. Every feature is either in scope or out of scope. "Out of scope" doesn't mean "no" — it means "we can build that in Phase 2". This single practice has saved more projects than any technical skill I have.

Technically: build with scope in mind. Use feature flags for work-in-progress features so you can ship without unfinished code visible to users.

Lesson 2: Test on Real Devices, Not Just Simulators

Simulators are convenient but deceptive. I've shipped features that worked perfectly in the simulator and broke immediately on real hardware.

Real-device bugs I've encountered that were invisible in simulators:

  • Camera permissions behaving differently on specific Android OEM builds (Samsung, Xiaomi)
  • Touch target areas being too small for real fingers
  • Biometric authentication (fingerprint/face ID) failing silently
  • Network transitions (WiFi → mobile data) causing unexpected state resets
  • Performance issues on older devices that run smooth on the latest iPhone simulator

Minimum device testing matrix I use before any major release: 1 current Android flagship, 1 mid-range Android (2-3 years old), 1 current iPhone, 1 older iPhone (e.g., iPhone 11).

Lesson 3: The Backend Will Let You Down — Build For It

In my first job, I assumed the API would always return what the documentation said. I was wrong. APIs return nulls where there should be strings, empty arrays instead of null, and status 200 with an error message in the body.

Build your app as if the API is unreliable:

  • Null-check everything from the API, even "required" fields
  • Use ?.toString() ?? '' defensively instead of direct property access
  • Validate API responses against your models — log mismatches to your crash reporter
  • Design your UI to handle empty states, partial data, and loading states for every screen
  • Never show a crash to the user — catch exceptions at the boundary and show a friendly error screen

The Realty Pro Shots app I built was particularly demanding — camera hardware + AWS S3 uploads + FCM notifications in sequence. One failure in that chain needed graceful recovery without data loss. Building defensively from day one made that achievable.

Lesson 4: Invest in Developer Tools Early

I resisted setting up proper tooling in early projects because "the app is small, it's not worth it". That thinking cost me hours of debugging time later.

The tooling I now set up on day one of every project:

  • Crash reporting — Firebase Crashlytics (free). You'll see crashes from users that you'd never catch yourself.
  • Analytics — Firebase Analytics. Know which screens users visit and which they abandon.
  • Logging — Structured logging with log levels. Filter to errors in production, verbose in development.
  • Linting — flutter_lints or ESLint with strict rules. Enforce code style so reviews focus on logic, not formatting.
  • Environment configuration — Separate dev/staging/production configs from day one. Never hardcode API URLs.

Lesson 5: Communication Is a Technical Skill

This was the hardest lesson. Early in my career, I thought my job was to write code. I now know that my job is to help clients solve problems — and code is just one of the tools.

The developers who advance fastest are the ones who can:

  • Explain technical constraints in plain language ("we can't do that in 2 days because...")
  • Push back on bad ideas respectfully with alternatives ("that would work, but here's a simpler approach")
  • Give accurate time estimates (and add a 30% buffer — always)
  • Proactively communicate blockers before they become problems
  • Document decisions so future-you (and teammates) understand the why

I've seen technically brilliant developers get stuck in junior roles because they couldn't communicate. And I've seen average developers advance quickly because clients trusted them completely.

Your code will be read by humans before it's executed by machines. Write for the humans first.

Bonus: The One Thing That Changed My Career

Build something real, outside of work, that solves a problem you have. It doesn't need to be groundbreaking. The discipline of taking a project from idea to store submission — on your own time, with no one pushing you — teaches you things no job can.

That discipline is what separates people who code for a salary from engineers who build for impact.
