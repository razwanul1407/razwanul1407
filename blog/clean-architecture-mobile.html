Clean Architecture for Mobile Apps

Md Razwanul Hasan

October 20, 2025  ·  10 min read

When I joined my first startup, the codebase was a mess — UI logic mixed with API calls, hardcoded strings everywhere, and no tests. Every feature took three times longer than it should. Since then, I've obsessed over structure. Here's what I've settled on.

Why Architecture Matters

A mobile app without architecture is a time bomb. It works fine with 5 screens. By screen 20, you're debugging spaghetti code at 2am. Clean Architecture solves this by separating concerns into layers where each layer has one job and doesn't know what's outside it.

The Three Layers

1. Presentation Layer (UI)

This layer contains your screens, widgets/components, and view models. Its only job is to display data and capture user input. It never calls the API directly. It never has business logic. It talks to the domain layer through use cases.

In Flutter, I use BLoC here. Each screen has a corresponding Bloc that takes events (user actions) and emits states (what the UI should show). In React Native, I use custom hooks + React Query.

// Flutter BLoC example
class OrderBloc extends Bloc<OrderEvent, OrderState> {
  final PlaceOrderUseCase placeOrder;

  OrderBloc({required this.placeOrder}) : super(OrderInitial()) {
    on<PlaceOrderRequested>((event, emit) async {
      emit(OrderLoading());
      final result = await placeOrder(event.orderData);
      result.fold(
        (failure) => emit(OrderError(failure.message)),
        (order) => emit(OrderSuccess(order)),
      );
    });
  }
}

2. Domain Layer (Business Logic)

This is the heart of your app. It contains:

  • Entities — Plain data objects (Order, User, Product). No framework dependencies.
  • Use Cases — Single-responsibility classes that execute one business operation.
  • Repository Interfaces — Abstract contracts that define what data operations exist. The domain layer doesn't know how data is fetched.

This layer has zero Flutter/RN dependencies. It's pure Dart or JavaScript. This makes it trivially testable.

3. Data Layer (Data Sources)

This layer implements the repository interfaces from the domain layer. It contains:

  • Repository Implementations — Concrete classes that implement domain interfaces
  • Remote Data Sources — API calls (Dio/Axios, REST)
  • Local Data Sources — SQLite, SharedPreferences, Hive, AsyncStorage
  • Models — JSON-serializable versions of domain entities

Folder Structure I Use

lib/
├── core/
│   ├── errors/          # Failures, Exceptions
│   ├── network/         # Dio client, interceptors
│   └── utils/           # Constants, formatters
├── features/
│   └── orders/
│       ├── data/
│       │   ├── datasources/
│       │   ├── models/
│       │   └── repositories/
│       ├── domain/
│       │   ├── entities/
│       │   ├── repositories/  (interfaces)
│       │   └── usecases/
│       └── presentation/
│           ├── bloc/
│           ├── pages/
│           └── widgets/
└── injection_container.dart  # Dependency injection

Dependency Injection

I use get_it in Flutter for dependency injection. This means every class receives its dependencies through its constructor — nothing is instantiated inside a class. This makes testing straightforward: just inject mock implementations.

Error Handling Pattern

I use Either<Failure, Success> from the dartz package (Flutter) or a custom Result type in React Native. Every use case returns an Either — forcing the UI layer to handle both success and failure cases explicitly. No silent failures.

Real-World Application

I applied this structure on BIFLE (truck booking, 2 companion apps) and Quality Group ERP. When new features were requested mid-development, adding them required touching only the relevant feature folder — nothing else broke. That's the payoff of architecture investment upfront.

Common Mistakes to Avoid

  • Putting API calls directly in a ViewModel/screen widget
  • Using the same model for both API responses and UI display
  • Storing business logic in the repository implementation instead of use cases
  • Skipping interfaces because "it's just one data source for now"
Architecture is not about following rules for their own sake. It's about creating the conditions where future-you (or your teammates) can work fast without fear.
