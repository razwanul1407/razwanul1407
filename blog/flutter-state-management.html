State Management in Flutter: BLoC vs Riverpod

Md Razwanul Hasan

June 5, 2025  ·  9 min read

I've used BLoC on large-scale enterprise apps and Riverpod on fast-moving product apps. Both are excellent. Here's when I reach for each — with real examples from apps I've shipped.

Why State Management Matters in Flutter

Flutter's reactive UI model is powerful, but setState() only works for local widget state. As soon as you need to share state across the widget tree — user authentication, cart items, network data — you need a proper solution.

The Flutter community has converged around two mature solutions: BLoC/Cubit and Riverpod. Both are production-ready. The choice comes down to team size, app complexity, and preference for explicitness vs conciseness.

BLoC (Business Logic Component)

BLoC enforces a strict pattern: the UI emits Events, the BLoC processes them and emits States. The UI rebuilds when state changes. This unidirectional data flow makes debugging straightforward — you always know what triggered a state change.

// Events
abstract class CartEvent {}
class AddToCartRequested extends CartEvent {
  final Product product;
  AddToCartRequested(this.product);
}

// States
abstract class CartState {}
class CartLoading extends CartState {}
class CartLoaded extends CartState {
  final List<CartItem> items;
  final double total;
  CartLoaded({required this.items, required this.total});
}
class CartError extends CartState {
  final String message;
  CartError(this.message);
}

// BLoC
class CartBloc extends Bloc<CartEvent, CartState> {
  final CartRepository repository;

  CartBloc({required this.repository}) : super(CartLoading()) {
    on<AddToCartRequested>(_onAddToCart);
  }

  Future<void> _onAddToCart(
    AddToCartRequested event,
    Emitter<CartState> emit,
  ) async {
    emit(CartLoading());
    final result = await repository.addItem(event.product);
    result.fold(
      (failure) => emit(CartError(failure.message)),
      (cart) => emit(CartLoaded(items: cart.items, total: cart.total)),
    );
  }
}

When BLoC Shines

  • Large teams where explicit patterns prevent inconsistency
  • Apps with complex state transitions (multi-step flows, real-time data)
  • When you want built-in logging/testing support from bloc_test
  • Enterprise apps where auditability matters

I used BLoC on BIFLE User & Driver (real-time location tracking, booking state machine) and Hillive (live streaming state management). The explicit event/state model was essential for debugging race conditions.

Riverpod

Riverpod is more concise. You define providers (pieces of state or computed values) and watch them in your widgets. No event classes needed for simple cases.

// Define providers
final cartRepositoryProvider = Provider<CartRepository>((ref) {
  return CartRepositoryImpl();
});

final cartProvider = AsyncNotifierProvider<CartNotifier, Cart>(() {
  return CartNotifier();
});

class CartNotifier extends AsyncNotifier<Cart> {
  @override
  Future<Cart> build() async {
    return ref.watch(cartRepositoryProvider).getCart();
  }

  Future<void> addItem(Product product) async {
    state = const AsyncValue.loading();
    state = await AsyncValue.guard(() async {
      return ref.read(cartRepositoryProvider).addItem(product);
    });
  }
}

// In widget
class CartScreen extends ConsumerWidget {
  @override
  Widget build(BuildContext context, WidgetRef ref) {
    final cartAsync = ref.watch(cartProvider);
    return cartAsync.when(
      data: (cart) => CartView(cart: cart),
      loading: () => const CircularProgressIndicator(),
      error: (err, _) => ErrorView(message: err.toString()),
    );
  }
}

When Riverpod Shines

  • Smaller teams or solo developers
  • Fast-moving product apps where boilerplate slows you down
  • Apps with many small, interconnected pieces of state
  • When you want compile-time safety for provider dependencies

The Practical Comparison

CriteriaBLoCRiverpod
BoilerplateMore verboseLess verbose

Learning CurveSteeperGentler
ExplicitnessVery explicitModerate
Compile safetyGoodBetter
Large teamsBetter (enforced patterns)Good

My Personal Choice

For enterprise or complex apps with a team: BLoC. The verbosity is a feature — it forces every developer to think explicitly about events and states, making code reviews easier and bugs more obvious.

For product apps or when moving fast: Riverpod. Fewer files, less ceremony, still fully testable.

The best state management solution is the one your team understands and uses consistently. A mediocre pattern used everywhere beats a perfect pattern used nowhere.
